# Аналитический отчёт: План доработки

## 1. ЧТО НУЖНО БЫЛО СДЕЛАТЬ (из «План реализации.md»)

План состоит из 4 этапов:

### Этап 1 — System Prompt
- **Backend:** добавить колонку `sysprompt` в таблицу `sessions`
- **Backend:** `getFullSession` возвращает `sysprompt`
- **Backend:** `updateSession` принимает и сохраняет `sysprompt`
- **Backend:** при формировании LLM-контекста — добавлять `system` сообщение из `sysprompt`
- **Frontend:** textarea в правой панели (вместо Statistics)
- **Frontend:** при отправке вопроса — сравнивать с предыдущим значением, при изменении — `PATCH`
- **Frontend:** при загрузке сессии — подставлять `sysprompt` из БД в textarea

### Этап 2 — Контекстное информирование
- **Backend:** FIFO-обрезка в `buildLLMContext`, возврат `excludedQAs` (массив индексов/ID)
- **Frontend:** мета-строка в ответе бота: *«Q1-Q3 были исключены из контекста…»* (курсив, серый, `.context-notice`)
- **Важно:** строка НЕ сохраняется в БД — только UI-представление

### Этап 3 — Чекбоксы (Standard vs Manual)
- **Standard:** все чекбоксы `checked` + `disabled`, Select All/Deselect All — `disabled`
- **Manual:** чекбоксы активны, пересчёт `contextLength` выбранных Q&A
- **UI:** переключатель режимов, бейдж «Context: Q5-Q15 (11/15)»

### Этап 4 — Интеграция
- LLM-запрос = `system prompt` + `выбранные Q&A` + `URL-контент` + `файлы` + `текущий вопрос`
- При превышении лимита — FIFO-обрезка + информирование

## 3. ВЫЯВЛЕННЫЕ НЕДОСТАТКИ (дополнительно к Плану)

### 3.1. Архитектурные проблемы (ваше замечание)

**Дублирование в `apiGateway.js`:**
- `sessions.getMessages(id)` → `GET /api/sessions/:id/messages` — история
- `sessions.getQAPairs(id)` → `GET /api/sessions/:id/qa-pairs` — сгруппированные пары

Это **не полное дублирование** (разные эндпоинты, разные данные), но:
- `getMessages` возвращает плоский список сообщений
- `getQAPairs` возвращает сгруппированные пары (только для Manual режима)
- **Проблема:** на фронте в `renderMessagesFromAPI` идёт работа с плоским списком, а для Manual режима нужен отдельный вызов `getQAPairs`. Это два источника правды.

**Рекомендация:** объединить — `getQAPairs` должен быть единственным источником для UI-списка вопросов (левая панель), а `getMessages` — только для рендера чата. Либо наоборот — сделать `getMessages` возвращающим сразу сгруппированные пары.

### 3.2. Проблема `retainPercent = 100` (из «Последний вопрос.js»)

- При 100% обрезка **никогда не срабатывает**, если история помещается в `contextSize * 4`
- Грубая аппроксимация `× 4` неверна для русского текста (реальное соотношение 3.5–4.5)
- **Результат:** при длинной истории модель получает переполненный контекст, но `removedQAPairs = 0` — выглядит как «всё хорошо»

### 3.3. Отсутствие фоллбэков

- Если `fixedChars > maxChars` → `availableChars = 0` → `targetChars = 0` → вся история обрезается, но **без предупреждения**
- Нет логирования итогового размера контекста в символах после `cutContext`

### 3.4. Порядок сообщений и «вес»

- Текущий вопрос — последний (правильно для трансформеров)
- Но если предыдущие Q&A пары большие, модель может «терять» текущий вопрос
- **Нет механизма приоритизации** текущего вопроса

### 3.5. Прочее

- В `script.js` есть опечатка: `windаow.apiGateway` (кириллическая `а`) в функции `returnToCurrent` — **баг, сломает возврат к текущему чату**
- В `style.css` есть закомментированные/дублирующиеся блоки (`.right-panel` описан дважды)
- `index.html` содержит только каркас — нет textarea System Prompt, чекбоксов, Mode dropdown
